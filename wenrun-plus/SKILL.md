# 文润 Plus

## 目标

把中文文本改到像中文作者自然写出的文章。保留原文事实、立场、信息密度、文体功能和必要结构，重点去掉翻译腔、AI 模板感、机械排比、空泛拔高、聊天机器人残留和过度格式化痕迹。

默认不改变事实边界，不补不存在的来源、数据、动机或背景。除非用户明确要求大改结构，否则尽量覆盖原文全部信息，并保留原有标题层级、段落功能和主要顺序。深度改写时可以合并、删减、挪动段落，但必须服务同一条主线。

## 工作顺序

1. 先判断原文真正要回答的问题，确认开头、主体、结尾是否沿着同一条主线推进。
2. 找出成簇出现的问题，不因一两个普通词误判。重点看翻译句法、三点式结构、空泛结论、格式装饰、聊天套话是否同时出现。
3. 先修文章主线，再修段落结构，最后修句子、词语和标点。
4. 根据文体决定改写强度。营销稿落回动作、成本和结果；评论和 newsletter 保留判断与声音；百科、技术文档和参考资料以清楚、直接、可核查为先。
5. 最后朗读一遍，确认句子推进自然，读起来不像翻译后的英文，也不像模型按模板生成的说明文。

## 改写原则

### 1. 保留事实，控制判断

- 不改变原文事实、时间、人物关系、因果关系和立场。
- 不轻易写「完全取代」「彻底结束」「必然改写整个行业」这类绝对判断。
- 趋势明显时，说清先影响谁、怎么影响、受什么条件限制。
- 证据不足时，缩小判断范围，不做阴谋论推断、资本市场臆测或人物动机脑补。
- 「专家认为」「业内人士指出」「多家媒体报道」「行业报告显示」必须有具体来源、时间或上下文；没有就删掉，或改成可核查事实。

### 2. 去翻译腔和英文句法外壳

优先处理这些写法：

- `对于……来说`、`基于……`、`围绕……展开`、`从某种意义上说`、`值得注意的是`、`与此同时`、`在这一背景下`、`使得……得以……`。
- 没有真实尺度关系的 `从 X 到 Y`。
- 被动句、无主句连用，如 `能力被进一步增强`、`结果会自动保存`、`无需猜测`。
- 为了避免重复而频繁替换同一个核心概念，导致指代不稳。

处理时能删就删，能换成短连接就换，能调整句序就不要硬加连接词。核心概念可以重复，不必为了显得高级而换同义词。教程、产品说明和技术文档里，主语和动作要写清楚。

**修改前**

> 在这一背景下，AI Agent 对于软件团队来说，正在成为提升效率的重要抓手。

**修改后**

> 在这样的环境里，AI Agent 正在进入软件团队的日常流程，先改变的是效率。

### 3. 删空话，把大词落到具体机制

警惕没有信息量的词和句式：

- `颠覆`、`革命`、`赋能`、`重塑`、`引领`、`开启新篇章`、`里程碑`、`深远影响`。
- `这标志着……`、`这意味着……的时代已经到来`、`真正的变革才刚刚开始`。
- `持续深化`、`全面推进`、`积极探索`、`取得显著成效`、`实现高质量发展`、`打造闭环`。

如果没有真实指标、动作或机制，直接删。需要强调重要性时，先写清对象、约束、成本、分工、流程或结果，再给判断。

**修改前**

> 这一能力将彻底颠覆内容产业，并为创作者生态带来革命性变化。

**修改后**

> 这一能力先改变的是内容团队的分工。过去需要外包或多人协作的初稿生产，现在可以在内部更快完成。

### 4. 打散机械结构

常见模板包括：

- `不是……而是……`、`不仅……还……`、`不再只是……而是开始……`。
- 每段都写成三分句、排比句或工整对照句。
- 全文反复使用 `首先 / 其次 / 最后`、`第一 / 第二 / 第三`、`第一层 / 第二层`。
- 每节都是「先定义，再拆 2 到 4 点，最后拔高一句」。
- 标题下面先来一句复述标题的空话，再进入正文。

处理方法：

- 对比关系不必总写成 `不是……而是……`，可以改成转折、递进或重心移动。
- 非教学型文本少用编号推进；能用时间、因果、场景或问题推进，就不要用目录式编号。
- 能写成自然段的列表，优先合并成段落。
- 保留必要的 bullet，但不要让每条都变成 `粗体小标题 + 冒号 + 解释`。
- 段落长短可以不齐，句子也不必每句同样长度。

**修改前**

> 这个平台带来了三点变化：第一，效率提升；第二，协作增强；第三，体验优化。

**修改后**

> 这个平台最先改掉的是工作方式。任务推进更快了，协作也更集中，很多原本零散的操作被收进了同一条流程里。

### 5. 管住文章主线和段落功能

文章级问题优先于词句问题：

- 开头要尽快立题，不要第一段说 A，后面一路滑到 B。
- 主体段落各自要有功能，常见功能是交代背景、提出判断、展开论据、举例、转折、收束。
- 某段既不推进主线，也不提供必要信息时，优先删、并、挪，不要硬留。
- 结尾要回应前文真正提出的问题，不要临时拔高到更大的时代命题。
- 总结结构时，不要写成一串 `先……再……最后……`，要看段落是不是顺着同一个问题自然往下走。

可采用的结构走法：

- **把问题钉住，再慢慢拆开**：适合行业分析、产品评论、趋势判断。先收窄题目，再讲机制、成本和边界。
- **先摆出现象，再给判断**：适合新闻评论、社会观察、产品现象分析。先让读者看见事情，再解释它说明什么。
- **从误判或经验写起**：适合个人成长、职业建议、newsletter。把转变、代价和适用条件讲明白。
- **先亮判断，但别说满**：适合评论、杂文、社论式文章。判断要准，后文用事实、反例和常识撑住。
- **先立住人物或案例，再外推**：适合人物评论、公司案例分析、纪实非虚构。结尾回到个案，不把案例吹成传奇。

这些是替代路径，不是模板。原文结构已经清楚时，只做局部修整；材料不足时，宁可缩短，不要硬撑完整骨架。

### 6. 处理结尾和段尾收束

两类结尾最容易显得像 AI：

- 口号式拔高：`未来已经到来`、`这只是开始`、`行业将迎来新的篇章`、`真正的变革才刚刚开始`。
- 预告式收束：`接下来要……`、`下一步是……`、`接下来的问题是……`。

处理时回到前文已经展开的变量、约束、代价或下一步具体变化。可以用冷结论、反向限制、开放问题或具体场景收束，不要把段尾写成时代宣言或下集预告。

**修改前**

> 这套机制目前已经在大模型推理里跑通了，接下来要解决的是它在端侧的部署问题。

**修改后**

> 这套机制目前已经在大模型推理里跑通了。但要把它压进手机和笔记本，参数量和功耗还差一个数量级。

### 7. 删除聊天机器人残留

删除这些服务型、路标式或过度顺从的句子，除非它们本来就是对话内容：

- `当然`、`没问题`、`你说得完全对`、`这是一个很棒的问题`。
- `以下是`、`希望这有帮助`、`如果你需要我继续`。
- `让我们深入探讨`、`下面是你需要知道的内容`、`闲话少说`。
- 文章开头直接宣布「本文将从三个方面展开」。

正文直接进入内容。教程可以有导语，但导语必须提供信息，而不是宣布自己要开始讲。

### 8. 保留人味，但不要表演人味

可以保留真实细节、旁白、犹豫、不整齐的句子和轻微不确定感。博客、随笔、评论、newsletter 可以有明确判断、短句和作者声音。

避免这些伪人味：

- 连续戏剧短句：`然后它出现了。没有预兆。没有退路。`
- 假坦诚开头：`说实话？`、`听着。`、`事情是这样的。`
- 格言公式：`X 是 Y 的语言`、`X 不是工具，而是一面镜子`、`效率成了陷阱`。
- 每段最后都落成一句可截图的金句。

短句可以用，但不要连着用来制造重量。判断要依附在事实、场景或经验上。百科、技术、法律和参考资料不必强行口语化，清楚、克制就是合适的人类声音。

### 9. 处理百科、资料和文档腔

百科和资料型文本里，少写价值判断，多写职责、时间、地点、数据和动作。

- 不把普通事实硬写成 `关键时刻`、`广泛趋势`、`深远影响`。
- 不用 `据认为`、`可能成长于`、`保持低调` 等句子填补资料空白。
- 公开资料缺失时，只写「现有公开资料没有记录……」，不要推断人物性格、隐私观或动机。
- 技术文档、注释和说明书默认描述当前系统，不要写成提交记录。除非文本本来是 changelog、迁移指南或发布说明，否则少用 `新增`、`以前`、`现在`、`本次更新`。

**修改前**

> 关于她早年经历的公开资料很少，这表明她一直保持低调，并重视个人隐私。

**修改后**

> 现有公开资料没有记录她的早年经历。

### 10. 统一标点、格式和术语

- 正文引号默认用全角双引号 `“”`，嵌套用全角单引号 `‘’`。如果项目约定或用户本轮明确要求使用 `「」`，全篇统一改用 `「」`，嵌套用 `『』`。两种样式不要混用。
- 不使用长破折号 `——`，优先改成逗号、句号或拆句。
- 不密集使用冒号 `：`，尤其避免连续多句都靠冒号展开解释。
- 去掉不必要的粗体、emoji、图标式 bullet；正文不要靠格式制造重点。
- 中文正文中的英文多词术语使用半角空格分词，如 `AI Design Agent`。
- 英文术语与中文括号连写时，不在 `（` 前加空格，如 `LLM（大语言模型）`。
- 并列英文术语用斜杠连接时，不在斜杠两侧加空格，如 `coworkers/agents`。
- 英文品牌名使用官方大小写，如 `YouTube`、`OpenAI`、`GitHub`。
- `token`、`API` 保持英文。
- `PR` 在长篇中文叙述里优先写作 `代码审查`，除非项目已有明确约定。
- 数字日期写作 `2026 年 2 月 26 日`、`2 月 5 日`；中文月份叙事写作 `一月`、`二月`。

## 常见改法速查

| 问题 | 优先改法 |
| --- | --- |
| 翻译腔连接词密集 | 删连接、换短句、调整句序 |
| 空泛大词和口号 | 写成具体动作、机制、成本、约束或结果 |
| 三点式和编号撑全文 | 改成时间、因果、场景或问题推进 |
| 列表过多 | 能合并就合并成自然段 |
| 段尾总拔高 | 回到前文变量、边界或具体后果 |
| 预告式收束 | 改成冷结论、限制条件或开放问题 |
| 模糊权威 | 写清来源；没有来源就删 |
| 粗体、emoji、标题热身 | 去格式装饰，让正文承担重点 |
| 伪人味 | 删假坦诚、金句和连续戏剧短句 |
| 文档 diff 腔 | 描述当前能力、输入、输出和约束 |

## 输出方式

默认按下面顺序交付：

1. 直接给改写后的版本。
2. 如果用户要求解释，再简短指出 3 到 6 个最影响阅读的问题。
3. 如果用户要求保留更多原句，再补一个「轻改版」和一个「重写版」。
4. 如果用户明确要求按检测流程交付，可给「草稿 → 仍像 AI 的问题 → 最终版」。默认不要把流程摊开。

## 最终检查

交付前确认：

- 读起来像中文作者在写，不像翻译后的英文。
- 主体段落都服务同一条主线，没有明显跑题或重复。
- 句子之间自然推进，不靠模板连接词硬粘。
- 判断强度和事实证据匹配。
- 没有聊天机器人套话、路标式开场、过度粗体、emoji 或格式装饰。
- 没有模糊权威、猜测式补空、无来源的知名度堆砌。
- 全文节奏没有连续重复「先定义、再拆项、最后拔高」。
- 标点、术语、日期、品牌大小写和引号样式全篇统一。
